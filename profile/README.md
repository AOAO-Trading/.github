# AOAO Trading

![schema](assets/aoao_trading.png)

## Service description
### API message reader and normalizer service
Implements GET requests and streams reading from crypto exchanges. As
different exchanges implement different request/data stream body, each service
must implement normalization logic. This service must have different
implementation for each exchange and is replicated for each symbol/currency
pair.
### Order Book Advisor Service
Retrieves current limit order book with GET request from API message reader and
sends the snapshot to the local order book of newly launched strategy service.
Saves the state of each separate order book into in-memory database.
### Strategy Service
Even though the following description of modules does not describe different
services as they do not use network based communication, those implement
different logic and responsibility thus we decided to describe them separately.
#### Local Order Book
When a strategy service is run it launches its own order book. Order book
snapshot is initialized with call to Order Book Advisor Service. After LOB being
initialized it starts receiving stream updates from API message reader and updates
order book state.
#### Strategy logic
On each order book update signal extraction is fired. Each Strategy contains a
logic that combines information given to it (limit order book, trades, statistics read
from Data Lake) and decides whether we should trigger buy/sell/ignore signal.
#### Risk Engine
Combines information obtained from Local Order Book, Monitoring and statistics
database to set appropriate take profit / stop loss limits. Also implements logic that
completely blocks signal from propagation.
### Execution Gateway
After passing all checks in risk engine this service gets an action signal with
parameters regarding position size, TP/SL, etc. This service verifies the action
signal with open position in order management system (OMS), if order
management system approves it the execution gateway formats the order and
sends to cryptocurrency exchange. After it sends the order to OMS, where it is
saved in “desired position” data base. After execution gateway receives message
that the order was accepted, it sends request to OMS to move this order from
“desired positions” to “active positions”. In case order was declined, execution
Architecture 2
gateway sends request to remove the order from desired/active position
respectively.
### Order Management System
Implements logic of writing/reading desired/active orders into in memory
database. Also, implements logic for checking whether order generated by
strategy is adequate. In case of order decline sends message request to
Monitoring service.
### Monitoring and Strategy disabler service
Monitors trading results: profit and losses. In case of stop loss condition triggered
disables respective strategies and writes activity logs into on-disk database. Uses
on-disk database as a config. Provides endpoints for user interface to see open
positions, trading results, etc
